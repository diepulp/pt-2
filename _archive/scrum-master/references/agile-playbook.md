# PT-2 Agile Playbook

## Context: Solo Developer + AI Agents

PT-2 is developed by a single developer (Vladimir) working with AI coding agents. This fundamentally changes what "agile" means compared to a team of 5-8 humans.

**What stays relevant:**
- **Visibility**: The board is the single source of truth for what's planned, active, and done
- **Traceability**: Every piece of work should connect to a ticket, and tickets should connect to commits/PRs
- **Prioritization**: Limited bandwidth means ruthless prioritization matters more, not less
- **Incremental delivery**: Ship in slices, not big-bang releases
- **Retrospection**: Regularly assess what's working and what's not

**What changes:**
- **No daily standups** (there's no team to sync). Instead: board status checks on demand
- **No sprint commitments** in the traditional sense. Work flows continuously, closer to Kanban
- **No story points** for team velocity. Effort estimation is about sequencing, not capacity planning
- **Ceremonies are async**: Planning, grooming, and retros happen in Claude Code conversations, not meetings

## Board Philosophy

### P2K (Kanban) — The Operational Board
This is the primary work-tracking board. It uses a simple To Do → In Progress → Done flow because the overhead of additional columns isn't worth it for a solo workflow.

**When to use P2K:**
- Technical tasks, tech debt, ops work
- Implementation work from build-pipeline execution
- Bug fixes and hardening
- One-off investigations

### SCRUM (Scrum Board) — The Feature Board
Used for feature development that benefits from the structure of stories, epics, and an In Review column.

**When to use SCRUM:**
- Feature development tracked by PRDs
- Work that has a design → implement → review lifecycle
- Larger initiatives broken into stories

## Issue Lifecycle

### Healthy flow:
```
PRD/Design → Epic created → Stories/Tasks decomposed
  → To Do (prioritized in backlog)
  → In Progress (actively being worked on)
  → In Review (PR open, waiting for merge) [SCRUM only]
  → Done (merged, deployed)
```

### Anti-patterns to watch for:
- **Zombie To Do**: Issues sitting in To Do for weeks — either prioritize them or archive
- **Phantom In Progress**: Multiple items "In Progress" simultaneously — in a solo workflow, realistically only 1-2 things are truly active
- **Missing Done**: Work merged but ticket still open — sync gap between code and board
- **Epic without children**: An epic with no decomposed stories/tasks is just a wish

## Prioritization Framework

Since PT-2 is a solo project, prioritization is about sequencing, not team allocation.

**Priority levels (map to Jira priority):**
- **Highest**: Blocking production or security issue — drop everything
- **High**: Critical path for current initiative — do next
- **Medium**: Important but not urgent — plan for it
- **Low**: Nice to have — do when there's slack
- **Lowest**: Someday/maybe — keep in backlog for reference

**Sequencing heuristic:**
1. Is anything on fire? (security, broken prod) → Do first
2. What has momentum? (active branch, recent commits) → Continue it
3. What unblocks the most? (dependency chains) → Clear the path
4. What's been neglected? (stale high-priority items) → Re-evaluate

## Labels as Taxonomy

Labels serve as a lightweight categorization system:

- **Domain labels**: `backend`, `security`, `measurement-ui` — what part of the system
- **Initiative labels**: `wedge-c` — which strategic initiative
- **Nature labels**: `hardening` — what kind of work

Use labels to filter the board by concern. Avoid label proliferation — if a label would only apply to 1-2 issues, it's not worth creating.

## Connecting Jira to PT-2 Artifacts

The PT-2 project has a rich documentation system. Jira issues should reference these artifacts:

| Artifact | Location | Jira connection |
|----------|----------|-----------------|
| PRD | `docs/10-prd/PRD-NNN-*.md` | Epic summary links to PRD |
| ADR | `docs/20-architecture/ADR-NNN-*.md` | Mentioned in issue description |
| EXEC-SPEC | Generated by build-pipeline | Linked in epic description |
| Issue docs | `docs/issues/` | Referenced in bug/task descriptions |
| Security docs | `docs/30-security/` | Referenced in security-labeled issues |

## WIP Limits

For a solo developer + AI workflow:
- **In Progress**: Max 2 items (one human-focused, one agent-delegated)
- **In Review**: Max 3 items (PRs waiting for merge)
- **To Do (prioritized)**: Top 5-7 items should be clearly ranked

Everything else in To Do is backlog — unranked and uncommitted.
