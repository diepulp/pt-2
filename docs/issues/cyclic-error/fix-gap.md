**Root Cause**
- INV-ERR-DETAILS requires every error payload to be JSON-safe and relies on `safeDetails()` as the “last line of defense” before anything is serialized. That guard only exists inside `lib/http/service-response.ts`, so it protects API routes that call `successResponse()` / `errorResponse()` but not other transports documented in the same standard.
- The server-action pipeline never runs that guard. The legacy helper (`lib/server-actions/with-server-action-wrapper.ts:46-147`) copies whatever `value.details` the handler returned straight into the `ServiceResult` envelope; only the `catch` branch sanitizes via `mapDatabaseError()`. The compositor pipeline (`lib/server-actions/middleware/tracing.ts:22-41`) likewise spreads `result.details` without touching it. When a handler returns a manual failure envelope (common in actions that want to preserve additional metadata) and includes a raw Supabase/PostgREST error object in `details`, Next.js tries to serialize that object when streaming the server-action response and crashes with `TypeError: cyclic object value`, so the operator never sees the human-readable code.
- Several API routes bypassed the guard as well. For example, `GET /api/v1/visits` returns `NextResponse.json(result, …)` when `withServerAction` fails (`app/api/v1/visits/route.ts:60-84`), and the rating-slip routes at `app/api/v1/rating-slips/[id]/move/route.ts:102-140` and `app/api/v1/rating-slips/[id]/modal-data/route.ts:354-413` rebuild their own JSON bodies by copying `result.details`. All three endpoints therefore re-expose the same cyclic-object failure if any upstream caller forgets to run `safeErrorDetails()`, which is precisely the discipline that regressed before the latest commit.

**Why the fix in 0af9114 wasn’t enough**
- That commit standardized raw-error extraction in the service layer and added `safeDetails()` to the HTTP helpers, but it never touched the server-action wrappers or the API handlers that shortcut straight to `NextResponse.json`. As a result, the seed issue reappeared as soon as one of the manual `ServiceResult` responses resurrected a raw Supabase error (the seed workflow hit exactly that path), proving that the boundary sanitizer must live in the transport wrappers, not just at the call sites.

**Recommended Remediation**
1. Mirror the HTTP guard inside both server-action wrappers (legacy and compositor): before returning either success or failure results, run `safeDetails()` on `result.details`. This guarantees every server action complies with INV-ERR-DETAILS even if a handler forgets to sanitize.
2. Update the API routes that bypassed the helper to reuse `errorResponse()` / `successResponse()`, or at least wrap `result.details` in `safeDetails()` when building custom responses. Doing so closes the gap for `GET /api/v1/visits`, `POST /api/v1/rating-slips/[id]/move`, and `GET /api/v1/rating-slips/[id]/modal-data`, which are the current offenders.
3. (Optional hardening) Keep the codebase-level discipline Gauss confirmed by adding a lint rule or small codemod that flags any future `details:` assignments lacking `safeErrorDetails()` or a literal primitive object. Even if someone backslides, the new boundary guard will absorb it, but static tooling will stop the regression before it ships.

Once those wrappers enforce the same guard as the HTTP layer, any future failure will yield the intended domain-specific error payload instead of another ambiguous “cyclic object value” crash.
