# Agent-4 — Loyalty Balance-Cache vs Ledger Split-Brain + Undefined Reversal Model

**Diagnostic agent:** split-brain-authority (read-only)
**Domain:** Loyalty (remediation-surface §2.3)
**Angle:** Balance-cache vs ledger-aggregate split-brain + undefined reversal/correction model
**Branch:** `epson`  ·  **Date:** 2026-06-21

---

## 1. Angle Summary

The single fact **"a player's current points balance"** is represented in **three** independent places that can silently diverge while all presenting under one name. The system declares an invariant — `current_balance MUST equal SUM(loyalty_ledger.points_delta)` — but enforces it **only by RPC discipline**. The advertised "drift detection" is a **detect-not-prevent** mechanism, and on inspection the materialized-view half of it is effectively **dead** (never refreshed, never read, no production cron). Separately, the append-only ledger declares a `reversal` reason code with **no RPC and zero call sites** — the correction model for an immutable ledger is **undefined**. Finally, the daily **liability snapshot** (a financial-reporting input) is computed by summing the **cache**, not the ledger truth — so any cache drift propagates directly into a money figure.

Provenance verdict on "current balance": **the cache (`player_loyalty.current_balance`) is the de-facto authority for every read surface, but the ledger SUM is the *declared* authority.** Two declared-equal stores, one enforced only by convention = **Aggregate Split-Brain**. The numbers are usually correct; the architecture provides no structural guarantee they stay so, and no live signal when they don't.

This is the textbook **aggregate split-brain**: two stores declared to own the same fact, equality enforced by convention, divergence invisible until an operator (or an auditor) acts on the wrong one.

## 4. Reversal / Correction Gap Analysis

**The hazard (L4-03) is the most acute structural finding in this angle.** The loyalty ledger is explicitly **append-only and immutable** — `loyalty_ledger_deny_update ON ... USING (false)` with the policy comment *"Use reversal entries for corrections"* (`20251213003000_...:224-229`). The system therefore **mandates** a reversal mechanism as the sole correction path. Yet:

- `reversal` exists as an **enum value** (`20251213003000_...:48`), a **DTO union member** (`dtos.ts:38`), and a **Zod schema value** (`schemas.ts:32`).
- It has **no RPC** (no `rpc_reverse_*`), **no service function**, and **zero call sites** anywhere in `services/`, `app/api/v1/loyalty/`, or `components/loyalty/` (grep confirmed).

**Consequence — the correction model is undefined, forcing provenance-corrupting workarounds:**
1. **`manual_reward` misuse** — staff issue a positive "service recovery" credit to paper over an erroneous debit; reason code now lies about provenance.
2. **`adjustment` as a catch-all** — the enum comments allow `adjustment` to be `+/-`, so a correction masquerades as an admin adjustment with no link to the original entry. There is no `original_ledger_id` FK or metadata-reference enforcement.
3. **Negative `redeem` / direct edits** — blocked by `deny_update`, but the absence of a sanctioned path pressures toward out-of-band DB fixes, which silently desync R2 (feeding L4-01/L4-02).

Because every correction workaround mutates **R1 row provenance and R2 balance** without a deterministic, auditable link to the entry being corrected, the undefined reversal model is the **upstream cause** of cache-drift risk. It is rated **S4** (authority/compliance hazard) because loyalty liability is a reported money figure (L4-04) and a casino comp ledger is audit-sensitive.

**Cure:** a single `rpc_reverse_ledger_entry(p_original_id, p_reason, p_idempotency_key)` that (a) validates the original exists and is reversible, (b) INSERTs a compensating row with `points_delta = -original.points_delta`, `reason='reversal'`, and `metadata.reverses = original_id`, (c) updates R2 atomically in the same transaction, and (d) is the **only** sanctioned correction path — with `adjustment`/`manual_reward` explicitly disallowed for corrections via SRL/policy. This converts an undefined lifecycle into a single-owner, fully-provenanced transition.

## 5. Open Questions

1. **Is `current_balance` worth keeping?** The weakest mechanism preserving the invariant is read-time derivation (`SUM(points_delta)` with the pagination index already present, `ix_loyalty_ledger_pagination`). Is the cache a real performance need, or accreted complexity that introduces the entire split-brain? If kept, it should be **trigger-maintained atomic with each INSERT**, not hand-maintained in each RPC.
2. **Why does the liability snapshot sum the cache (L4-04)?** Was this a deliberate "snapshot what operators see" choice, or an oversight? It directly couples a money figure to the non-authoritative store.
3. **Is the dead MV (R3b) intended to be wired, or suppressed?** The comment promises a "cron for production" that does not exist. Either commit to the detector (scheduled REFRESH + alert) or remove it so it stops advertising a guarantee it doesn't deliver.
4. **What is the correction authority/role?** Reversal RPC role gate — admin-only (like reconcile), or pit_boss? And does a reversal of a `redeem` (comp already fulfilled/printed) require a void of the physical comp (cross-boundary to printing/fulfillment)?
5. **Does any retroactive valuation change (PRD-053 `cents_per_point`) re-stamp historical `balance_after`/liability?** Cross-reference the valuation angle: `balance_after` is points (unit-stable), but `estimated_monetary_value_cents` is valuation-dependent and snapshotted — a later valuation change makes prior snapshots non-comparable.
6. **Promo/accrual propagation split (remediation-surface §2.3):** accrual writes ledger+cache but emits **no `loyalty_outbox`** (grep confirms zero outbox refs in accrual RPC); promo emits outbox but no ledger. Out of this angle's scope but compounds L4-01 — confirm with the propagation-angle agent before any unification.
