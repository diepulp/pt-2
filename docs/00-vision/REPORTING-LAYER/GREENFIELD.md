## What a Meaningful Shift Report Would Contain

A shift report sanitized for upstream stakeholders (casino GM, corporate, regulators) would assemble data from 5-6 current surfaces into a single document:

**Header:** Casino name, gaming day, shift (swing/day/grave), pit boss on duty, report generated by, timestamp.

**Section 1 — Financial Summary:**
Per-table row: table name, game type, drop, fills, credits, win/loss, hold %. Casino-level totals row. Source: shift metrics + rundown data.

**Section 2 — Rating Activity:**
Total rated sessions, unrated sessions, rating coverage %. Per-table coverage breakdown. Average bet range across tables. Total theo generated. Source: shift dashboard + measurement coverage widget.

**Section 3 — Compliance:**
MTL triggers (count, patron names, aggregate volumes). CTR triggers (count, amounts). Source: compliance dashboard gaming day summary.

**Section 4 — Anomalies & Exceptions:**
Alerts fired (count by severity). Alerts acknowledged (count, by whom, notes). False positives flagged. Unacknowledged alerts remaining. Source: admin alerts + alert quality telemetry.

**Section 5 — Baseline Quality:**
Tables with opening snapshot (count, %). Provenance breakdown (how many from prior-day close vs. manual entry vs. fill observation). Source: shift dashboard coverage bar.

**Section 6 — Loyalty Liability:**
Outstanding points, estimated dollar liability, player count, valuation policy version. Source: loyalty liability snapshot.

## What It Would Take to Build

There are two layers to the gap:

### Layer 1: Report Data Aggregation API (1-2 days)

A single BFF endpoint — something like `GET /api/v1/reports/shift-summary?gaming_day=2026-03-27&shift=swing` — that calls existing service methods in parallel and assembles the response:

| Data Source | Already Has Service Method? | New Code Needed? |
|------------|---------------------------|-----------------|
| Shift metrics (per-table financials) | Yes — `rpc_shift_table_metrics` | Thin wrapper only |
| Closed sessions count | Yes — existing query in closed sessions panel | Thin wrapper only |
| Compliance summary (MTL/CTR counts) | Yes — `mtl_gaming_day_summary` view | Aggregation query |
| Alert summary (counts, ack rate) | Yes — `rpc_get_alert_quality` | Already exists |
| Active alerts list | Yes — `rpc_get_anomaly_alerts` | Already exists |
| Rating coverage | Yes — `measurement_rating_coverage_v` | Already exists |
| Audit correlation health | Yes — `measurement_audit_event_correlation_v` | Already exists |
| Loyalty liability | Yes — `loyalty_liability_snapshot` table | Direct read |
| Baseline provenance breakdown | Partially — shift metrics returns `opening_source` | Count aggregation |

Almost every data source already has a service method or queryable view. The aggregation endpoint would call them in parallel and return a typed `ShiftReportDTO`.

### Layer 2: Report Rendering & Export (2-3 days)

| Capability | Approach | Effort |
|-----------|---------|--------|
| **On-screen report** | New page at `/admin/reports/shift-summary` with date/shift selector. Renders the DTO as a structured document layout. | 1 day |
| **Print** | CSS `@media print` stylesheet on the report page. `window.print()` triggered by button. No library needed. | 0.5 day |
| **PDF** | Server-side: render report HTML → PDF via headless browser or `@react-pdf/renderer`. Client-side: `jsPDF` or browser print-to-PDF. | 1 day |
| **CSV** | Flatten financial summary table to CSV. `Blob` + `URL.createObjectURL` + download link. | 0.5 day |

### Total Estimate: 3-5 days

The data layer is ready. The gap is assembly and rendering. No new RPCs, no new migrations, no new security model — just a BFF endpoint that calls existing services and a report page that renders the result in a print-friendly layout.
