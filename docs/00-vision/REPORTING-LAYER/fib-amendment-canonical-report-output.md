# FIB Amendment Note — Canonical Shift Report Document Output and Email Distribution

**Target artifact:** `FIB-H-SHIFT-REPORT.md`  
**Amendment purpose:** align the Shift Report intake with the clarified requirement that the report serve as both a tangible handoff artifact and a digital document distributed by email to involved parties.  
**Status:** proposed amendment for splice before PRD / EXEC generation.

---

## Amendment Summary

The current FIB treats the Shift Report as:

- an on-screen report page
- a browser-printable artifact
- a bounded CSV export surface

That framing is no longer sufficient.

The clarified requirement establishes that the Shift Report will also function as:

- a **standardized digital document**
- a **circulated artifact sent over email**
- a **stable management / stakeholder report**, not merely a printed web page

This changes the feature’s output model.

The Shift Report must therefore be reinterpreted as having three distinct representations:

1. **On-screen review surface** — operator preview and inspection
2. **Canonical document output** — standardized, stable PDF artifact
3. **Distribution path** — manual email sending of that canonical artifact to involved parties

---

## Why Amendment Is Required

The current FIB explicitly assumes:

- browser print via CSS `@media print`
- no server-side PDF generation
- no automated report scheduling or email delivery

That posture was acceptable only while the report was treated as an operator convenience artifact.

Once the report is expected to circulate digitally as an official copy, browser print becomes inadequate as the canonical output path because it does not guarantee:

- stable formatting across environments
- controlled pagination
- standardized report identity
- predictable archival quality
- consistent attachment output for email circulation

Therefore:

> **Browser print may remain as an operator convenience path, but it must no longer be treated as the authoritative artifact path.**

---

## Governing Decision

The Shift Report shall adopt the following output hierarchy:

### 1. On-screen report page
Used for operator review, validation, and local decision-making.

### 2. Canonical PDF document
Used as the authoritative report artifact for printing, attachment, archival, and management circulation.

### 3. Email distribution
Used to send the canonical PDF manually to the intended stakeholders.

### 4. Browser print
May remain as a convenience fallback, but is not the canonical artifact path.

---

## Required FIB Changes

## 1. Reclassify the report output model

The FIB must no longer imply that `window.print()` / browser print is the primary output model.

### Mandated replacement
Replace the implicit “render page then print browser output” posture with:

> **The on-screen report page is a review surface. The canonical Shift Report artifact is a standardized PDF generated from a fixed document template. Browser print may remain available as a convenience fallback but is not authoritative.**

---

## 2. Amend internal workstream partition

The current two-lane model should become three lanes, or remain two lanes with WS2 explicitly expanded. The cleaner option is three lanes.

### Proposed partition

#### WS1 — Report Assembly
Construct a read-only `ShiftReportDTO` from existing service outputs only.

Allowed work:
- field mapping
- section composition
- totals reconciliation
- snapshot assembly
- read-only aggregation using existing accepted data sources

Not allowed:
- new report-only business rules
- schema changes
- new RPC/view creation unless formally amended

#### WS2 — Canonical Report Rendering
Render the DTO into:
- on-screen review surface
- fixed report template
- canonical PDF artifact

Allowed work:
- standardized layout
- controlled pagination
- template styling
- print-safe formatting

Not allowed:
- configurable layouts
- multi-template support
- free-form report builder behavior

#### WS3 — Manual Distribution
Support manual send / attachment workflow for the canonical PDF to involved parties.

Allowed work:
- manual send action
- defined recipient classes
- attachment of canonical PDF
- audit-safe sending workflow if already compatible with project email posture

Not allowed:
- scheduled sends
- rules engine for routing
- recurring automation unless separately scoped

---

## 3. Introduce a fixed Shift Report template standard

The FIB must explicitly define that the report is not a generic printed page, but a standardized document.

### Mandated standard
Adopt a **Shift Report Standard Template v1** with fixed section order.

Recommended sections:

1. **Report Header**
   - casino name
   - property / pit
   - gaming day
   - shift boundary
   - generated timestamp
   - generated by
   - report reference ID
   - internal / confidential marking

2. **Executive Summary Strip**
   - total drop
   - total fills
   - total credits
   - total win/loss
   - hold %
   - total theo
   - rating coverage %
   - anomaly count
   - compliance trigger count

3. **Section 1 — Financial Summary**
   - per-table financial rows
   - totals row

4. **Section 2 — Rating / Coverage Quality**
   - coverage metrics
   - rated/unrated summary
   - quality flags

5. **Section 3 — Compliance Summary**
   - MTL / related summary indicators
   - summary-level only unless separately approved

6. **Section 4 — Exceptions and Anomalies**
   - counts
   - unresolved items
   - action-needed summary

7. **Section 5 — Baseline / Evidence Quality**
   - provenance summary
   - audit correlation quality
   - evidence quality statement

8. **Section 6 — Loyalty Liability**
   - outstanding points
   - estimated valuation
   - policy version

9. **Footer / Attestation Block**
   - generated by
   - optional reviewed by
   - page numbering
   - report reference
   - point-in-time disclaimer

---

## 4. Define document sanitation requirements

The FIB should add a sanitation rule to prevent a dashboard page from masquerading as a report.

### Mandated sanitation rule
The canonical Shift Report document must be:

- fixed in section order
- fixed in typography and spacing
- free of navigation chrome
- free of interactive controls
- free of screen-responsive layout dependency
- controlled for pagination
- suitable for print and PDF archival
- reviewed for sensitive-data exposure appropriate to the recipient class

This is a document template, not a raw printed web screen.

---

## 5. Replace the current print outcome language

The current FIB emphasizes browser print.

### Mandated replacement
Replace language like:
> “Print produces a professional, readable document…”

with:

> **Canonical PDF generation produces a professional, standardized, circulation-ready Shift Report suitable for printing, email attachment, and archival. Browser print remains available only as an operator convenience fallback.**

---

## 6. Amend exclusions around PDF and email

The current FIB excludes:
- server-side PDF generation
- automated report scheduling or email delivery

These exclusions must be revised carefully.

### Revised position
- **Remove the blanket exclusion on canonical PDF generation**
- **Replace blanket email exclusion with manual distribution only**
- Continue to exclude:
  - scheduled delivery
  - recurring delivery automation
  - distribution rules engines
  - multi-recipient policy logic beyond a simple bounded send workflow

### Mandated phrasing
> **Canonical PDF generation is in scope. Manual email distribution of the canonical Shift Report PDF is in scope. Scheduled delivery, recurring distribution, and automated routing remain out of scope for MVP.**

---

## 7. Add recipient-boundary guidance

Because the report will circulate, the FIB should specify who it is for.

### Recommended recipient classes
- incoming supervisor
- GM
- finance stakeholder
- compliance stakeholder where appropriate

### Mandated guardrail
If different recipient classes require different content redaction or detail levels, that is **not** to be solved in MVP by multiple templates unless explicitly amended. MVP uses one standard template unless a compliance review proves that a single-template model is unsafe.

---

## 8. Update expansion trigger rule

The expansion rule must reflect the new artifact model.

### Add triggers for amendment if:
- multiple report templates are proposed
- redaction variants by recipient are proposed
- scheduled email delivery is proposed
- automatic recipient resolution is proposed
- persistent report archival records or document registries are proposed
- signing / certification workflows are proposed

## Suggested Patch Language

The following language is suitable for direct splice.

### Replace / extend internal workstream partition

> **WS2 — Canonical Report Rendering**  
> Render the assembled `ShiftReportDTO` into both an on-screen review surface and a fixed Shift Report Standard Template v1 suitable for canonical PDF generation. This lane includes standardized layout, controlled pagination, and print-safe formatting. It does not include configurable layouts, template variants, or report-builder behavior.
>
> **WS3 — Manual Distribution**  
> Support manual distribution of the canonical Shift Report PDF to involved parties via the project’s approved email pathway. This lane includes bounded attachment sending only. It excludes scheduled delivery, recurring automation, routing rules, and recipient-policy engines.

### Add to required outcomes

> The Shift Report is available as a canonical, standardized PDF suitable for printing, email attachment, and archival.  
> The on-screen report page acts as a review surface; it is not itself the authoritative report artifact.  
> Manual email distribution of the canonical Shift Report PDF is supported for involved stakeholders.

### Add to exclusions

> Scheduled email delivery, recurring distribution automation, multi-template support, recipient-specific template variants, and automatic routing logic are excluded from MVP.

### Add sanitation rule

> The canonical report must use a fixed template with controlled pagination, standardized layout, no dashboard chrome, and no interactive UI residue. It must be suitable for both printed and emailed circulation as a stable management artifact.

---

## Alignment Verdict

This amendment changes the Shift Report from:

- a review page with browser print

into:

- a standardized reporting document workflow with bounded digital circulation

That is the correct shape if the artifact is expected to leave the app and be consumed by managers and other involved parties.

---

## Final Directive

**Re-evaluate `window.print()` as a convenience tool, not the canonical output path.**  
**Standardize the Shift Report as a fixed-template PDF artifact.**  
**Allow manual email distribution of that artifact.**  
**Keep scheduling, automation, routing, and template proliferation out of scope.**
