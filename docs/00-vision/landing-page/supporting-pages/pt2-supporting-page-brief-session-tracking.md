# PT-2 Supporting Page Brief — Session Tracking

**Date:** 2026-04-10  
**Purpose:** Define the content direction for the supporting page **Session Tracking**.

This artifact exists to keep the page grounded in implemented system reality and to prevent the page from collapsing into either a Player 360 promo page or a rating-slip feature dump.

---

## 1. Role of This Page

The supporting-page architecture expands one operating loop at a time.

**Session Tracking** is the page that explains how the system keeps one coherent record of play as a player moves through the floor.

Its job is to answer:

> How does the system track a player session from arrival to closeout without breaking continuity?

This page should not read like clerical check-in software, and it should not become a generic profile page.

---

## 2. Core Promise

**Track the session, not just the seat.**

This is the page’s central claim.

The page should frame Session Tracking as the operating loop that connects:

- player arrival
- table assignment
- rating in progress
- pauses and interruptions
- table moves
- session continuation
- closeout

The strength of the page is continuity.

Not data storage.  
Not forms.  
Not profile vanity.

---

## 3. Domain Boundary

Session Tracking, in system terms, is primarily composed of:

- instant player search
- visit check-in / check-out
- rating slips
- average bet capture
- pause / resume
- table moves
- session continuation
- start from previous session
- unidentified player tracking
- Player 360 as supporting context

If a section does not strengthen one of those ideas, it likely does not belong here.

---

## 4. What the Page Is Not

This page is **not** primarily about:

- floor-wide monitoring and shift metrics
- cashier workflows
- compliance thresholds
- onboarding, bootstrap, or import
- loyalty program mechanics
- table custody gates and session lifecycle as a primary topic
- full identity/compliance policy detail

Those can be referenced when necessary, but they are not the subject of this page.

---

## 5. Audience

Primary audience:

- pit managers
- floor supervisors
- operators responsible for tracking active play

Secondary audience:

- administrators evaluating whether the system can preserve operational continuity across real casino movement

This page should feel operational, not administrative.

---

## 6. Page Narrative

The page should follow a simple live-floor sequence:

1. Find the player
2. Start or continue the visit
3. Assign a table and begin rating
4. Pause, resume, or move the player without losing continuity
5. Close the session with the record intact

This is the right storytelling mode.

Not “look at our forms.”  
Not “here are all the profile tabs.”  
Not “here is a glossary of rating functionality.”

---

## 7. Primary Content Pillars

## 7.1 Start with the Player

This is the opening block.

The system supports:

- instant player search
- recent players access
- Player 360 profile access
- visit check-in / check-out
- enrollment and identity context where needed

This section should make clear that Session Tracking starts with quickly locating the player and anchoring the session correctly.

### Page purpose of this pillar

To establish that operators do not begin from a blank form.

They begin from the player, the current visit context, and the operating floor state.

---

## 7.2 Rate Live Play

This is the operational core.

The system currently supports:

- creating rating slips
- pausing rating slips
- resuming rating slips
- closing rating slips
- capturing average bet
- calculating theoretical win

This section should frame rating as part of live session handling, not as an isolated calculation engine.

### Page purpose of this pillar

To show that the system supports active session management while play is happening.

---

## 7.3 Preserve Continuity Across Interruption

This is the strongest differentiator on the page.

Real floor operations are messy:

- players move tables
- players step away
- sessions pause
- sessions resume
- play continues across interruptions

The system supports:

- pause / resume
- table moves with session continuity
- start from previous session
- visit continuation
- unidentified player handling when needed

This is where the page becomes materially more interesting than generic “visit tracking.”

### Page purpose of this pillar

To show that the product tracks the continuity of play, not just isolated table moments.

---

## 7.4 Keep Context Attached to the Player

Player 360 belongs here as supporting proof, not as the page’s subject.

Relevant context includes:

- session history
- financial activity
- loyalty activity
- summary metrics
- interaction timeline

The point is not “look at our profile.”
The point is:

> the session record stays attached to the player over time.

### Page purpose of this pillar

To show that active session handling and longer-term player context are connected.

---

## 7.5 Handle Imperfect Floor Reality

The page should acknowledge that not every session begins neatly.

The system supports:

- unidentified player tracking
- exclusions and player-status context
- returning to prior context where applicable

This section should remain practical and restrained.

Do not let it sprawl into a compliance or policy page.

### Page purpose of this pillar

To show that the system can handle real operational mess without breaking the record.

## 9. Tone Guidance

The tone of this page should be:

- operational
- current-tense
- restrained
- specific
- continuity-focused

## 12. Success Criteria

The Session Tracking page is successful if:

1. it is understandable in under 15 seconds
2. it feels like a live operating loop, not a form workflow
3. it clearly centers continuity across pauses, moves, and resumed play
4. it connects session handling to player context without becoming a profile page
5. it leaves adjacent loops to carry their own complexity

---

## 13. Final Summary

**Session Tracking** should establish one clean idea:

> The system keeps a player’s session coherent from arrival through rating, interruption, movement, and closeout, so operators track the continuity of play instead of patching together disconnected table moments.

That is enough.

Do not let the page swell into a second Player 360 page.
