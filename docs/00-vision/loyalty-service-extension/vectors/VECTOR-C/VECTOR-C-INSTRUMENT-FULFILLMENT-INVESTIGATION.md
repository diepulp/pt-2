## Vector definition

This vector covers everything required to produce a printable physical artifact when a pilot reward is issued — the constrained print standard, per-family templates, rendering mechanism, print trigger, print feedback, and minimal reprint support.

**Scope:** pilot reward instrument print standard, comp slip template, coupon template, iframe print utility, print trigger per issuance flow, print success/failure feedback, minimal reprint.

**Not in scope:** admin catalog CRUD (Vector A), issuance logic or persistence (Vector B), generalized print/document platform.

## Fulfillment classification (from REWARD_FULFILLMENT_POLICY.md)

The system recognizes three fulfillment families. Only the first two require printing:

| Family | Mechanism | Artifact | Print Policy |
|--------|-----------|----------|--------------|--------------|
| `points_comp` | ledger debit → comp record | **comp_slip** | **Required** |
| `entitlement` | coupon issuance record | **coupon** | **Required** |
| `internal_adjustment` | admin-only record | none | **Not required** |


### Critical structural rule

> Comp slips and coupons must NOT be collapsed into one domain model.

They may share a constrained print rendering path at the fulfillment layer, but:
- Storage and issuance semantics remain distinct
- Templates are per-family (comp slip ≠ coupon)
- Domain meaning is preserved through the print path

## What must be built (full scope)

### 1. Print rendering mechanism

**Approach:** Hidden iframe + `window.print()` (browser print dialog)

**Implementation outline:**
1. Button click (user gesture — required for `print()` to work)
2. `IssuanceResultPanel` assembles `FulfillmentPayload` from issuance result + context props
3. `onFulfillmentReady(payload)` fires — **this is Vector C's entry point**
4. Build printable HTML string from payload (all fields pre-resolved, no additional API call)
5. Create hidden iframe, write HTML into iframe document
6. Wait for readiness (fonts + 2x `requestAnimationFrame`)
7. Call `iframeWindow.print()`
8. Cleanup iframe after print dialog closes (best-effort)

**Location:** `lib/print/` (new directory)

**Files needed:**
- `lib/print/iframe-print.ts` — iframe creation, write, wait, print, cleanup utility
- `lib/print/templates/comp-slip.ts` — comp slip HTML builder
- `lib/print/templates/coupon.ts` — coupon HTML builder
- `lib/print/types.ts` — re-export `FulfillmentPayload` union from `@/services/loyalty/dtos` + `PrintResult` type
- `lib/print/templates/shared-styles.ts` — shared print CSS string (system fonts, `@page` margins)

### 2. Per-family templates

**IMPORTANT: Values are in integer cents.** All monetary fields in the fulfillment payload use `_cents` suffix (e.g., `face_value_cents: 2500` = $25.00). Templates must divide by 100 for display. The existing `formatCents()` helper in `issuance-result-panel.tsx:47` demonstrates the pattern: `` `$${(cents / 100).toFixed(2)}` ``.

#### Comp slip template (`points_comp` family)

**Required fields (from `CompFulfillmentPayload`):**
| Field | Payload key | Notes |
|-------|-------------|-------|
| Casino name | `casino_name` | Pre-resolved string |
| Player name | `player_name` | Pre-resolved string |
| Reward name | `reward_name` | From catalog (e.g., "Meal Comp", "Beverage Comp") |
| Reward code | `reward_code` | Short catalog code |
| Face value | `face_value_cents` | **Integer cents — divide by 100 for display** |
| Points redeemed | `points_redeemed` | Absolute value (positive integer) |
| Balance after | `balance_after` | Post-debit balance (integer points) |
| Issued by (staff) | `staff_name` | Pre-resolved string |
| Issued date/time | `issued_at` | ISO 8601 string |
| Reference number | `ledger_id` | UUID from `loyalty_ledger.id` |

**Layout:** Simple, generous spacing. Casino header → player info → comp details → staff/timestamp footer.

#### Coupon template (`entitlement` family)

**Required fields (from `EntitlementFulfillmentPayload`):**
| Field | Payload key | Notes |
|-------|-------------|-------|
| Casino name | `casino_name` | Pre-resolved string |

| Issued date/time | `issued_at` | ISO 8601 string |

**Layout:** Simple, generous spacing. Casino header → player info → coupon details → validation number (prominent) → staff/timestamp footer.

### 3. Print trigger integration

**Vector B has shipped the trigger infrastructure.** No mutation hooks need modification.

| Family | Trigger mechanism | What fires |
|--------|------------------|------------|
| `points_comp` | `IssuanceResultPanel` assembles `CompFulfillmentPayload` | `onFulfillmentReady(payload)` via `queueMicrotask` (auto) + Print button (manual) |
| `entitlement` | `IssuanceResultPanel` assembles `EntitlementFulfillmentPayload` | `onFulfillmentReady(payload)` via `queueMicrotask` (auto) + Print button (manual) |

**Trigger chain (verified):**
```
IssueRewardButton.onFulfillmentReady
  → IssueRewardDrawer.onFulfillmentReady
    → IssuanceResultPanel.onFulfillmentReady
      → assembleCompFulfillment() or assembleEntitlementFulfillment()
        → payload dispatched to callback
```

**Auto-fire behavior:** Fresh issuances (not duplicates) automatically fire the callback via `queueMicrotask()` on render. Duplicates (`isExisting: true`) only fire on explicit Print button click.

**Implementation:** Vector C provides the `onFulfillmentReady` handler that receives `FulfillmentPayload`, discriminates on `.family`, routes to the correct template builder, and calls the iframe print utility.

### 4. Print feedback

- **Success:** Print dialog opens → operator uses native browser print → dialog closes → UI shows "Printed" confirmation
- **Failure:** If iframe creation or `print()` call fails → UI shows error toast with "Print failed — try again" + manual retry button
- **No blocking:** Print logging (GAP-C3) must not block the print dialog

### 5. Minimal reprint

- Reprint the same coupon without creating a new issuance
- Reprint comp slip from history view (re-renders from persisted data)
- Reprint appends to `print_history[]` in metadata

## Gap analysis

### GAP-C0: `promo_coupon` table missing `metadata` column (P2 — downgraded)

**Impact:** Cannot store print history, channel tracking, or tier-at-issuance metadata.

**Finding:** Codebase investigation confirmed `promo_coupon` table (migration `20260106235611`, lines 91-115) has 20 columns but **no `metadata jsonb` column**. Latest migration timestamp is `20260319195353`.

**Fix:** New migration adding `metadata jsonb NOT NULL DEFAULT '{}'::jsonb` to `promo_coupon`. Then `npm run db:types-local` to regenerate types. Update promo service CRUD mappers to handle metadata field.

**Priority downgrade rationale:** This column is required for **print history logging** (GAP-C3, P2) but does NOT block core printing (GAP-C1, GAP-C2). Core printing consumes `FulfillmentPayload` which is fully assembled without metadata. Migration can ship in parallel with or after print utility/templates.

**Blocking:** Print history logging only. Core printing (utility + templates + trigger wiring) can proceed without this column.

### GAP-C1: No iframe print utility (P0)

**Impact:** Nothing can print. This is the foundation.

**Build scope:** ~80-120 lines. Hidden iframe creation, HTML injection, readiness wait, `window.print()` call, cleanup.

**Design constraints:**
- Must work with browser print dialog (no silent/background printing)
- System fonts only (no web font loading)
- Explicit `@page` margins
- Tolerant of 95-105% scaling
- No navigation (iframe stays hidden)

### GAP-C2: No per-family templates (P0)

**Impact:** Even with print utility, no content to render.

**Build scope:** ~100-150 lines per template (comp slip + coupon = ~200-300 lines total).

**Design constraints:**
- Large amounts, generous spacing
- No complex flex/grid edge cases
- No fragile absolute positioning
- Monospaced validation numbers
- Casino header with property name

### GAP-C3: No print history logging (P2)

**Impact:** No audit trail for prints. Acceptable for pilot with manual workaround.

**Build scope:** Best-effort append to `promo_coupon.metadata.print_history[]`:
```json
{
  "printed_at": "2026-03-18T10:25:00Z",
  "printed_by_staff_id": "uuid",
  "channel": "rating_slip_print_button",
  "device_hint": "Chrome/Windows"
}
```

**Important:** Print logging must NOT block printing. Fire-and-forget after print dialog opens.

### GAP-C4: No comp slip reference number system — RESOLVED

**Status:** Resolved by implementation reality. `CompFulfillmentPayload.ledger_id` (UUID from `rpc_redeem()`) serves as the traceable reference. The `CompIssuanceResult.ledgerId` is already surfaced in the issuance response and included in the frozen fulfillment payload.

**Decision:** Use `ledger_id` (UUID). While not human-readable, it is unique, immutable, and already in the payload. A short-format reference can be derived at the template layer if needed (e.g., display last 8 chars).

### GAP-C5: QR/barcode decision (P3)

**Impact:** Optional scan capability for validation numbers on coupons.

**Recommendation from posture audit:** Defer unless floor benefits from scan speed. Not required for pilot.

## Print standard specification (from governing boundary doc)

> Build the smallest print surface that makes every pilot instrument type usable on the floor. Stop there.

### What the print standard IS:
1. One rendering path (hidden iframe + `window.print()`)
2. Per-instrument templates (comp slip + coupon)
3. Common fields across all prints (casino, player, date, staff, reference)
4. Instrument-specific fields per template
5. Print trigger per issuance flow
6. Print success/failure feedback
7. Minimal reprint (manual workaround acceptable)

### What the print standard is NOT:
- Not a document management system
- Not a template registry with versioning
- Not a printer fleet abstraction
- Not a generic rendering engine
- Not designed for extensibility beyond the pilot instrument set
