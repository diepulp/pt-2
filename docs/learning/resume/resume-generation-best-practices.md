## 1. Resume Formats

### Three Standard Formats

| Format | Best For | ATS Friendliness |
|---|---|---|
| **Reverse-Chronological** | Steady work history, most roles | Highest |
| **Functional/Skills-Based** | Career changers, large employment gaps | Low — raises recruiter flags |
| **Hybrid/Combination** | Mid-senior, tech, creative roles | High if formatted cleanly |

**Default recommendation:** Reverse-chronological for 90% of candidates. It is the most universally expected and ATS-compatible structure.

### Page Length

- 1 page: < 5 years experience or entry-level
- 2 pages: 5–15 years experience (standard)
- 3 pages: Executive, academic, or highly technical roles only

### Visual Hierarchy

- Clear section breaks with bold headers
- Consistent font, size, and spacing throughout
- Adequate white space — recruiters scan in 6–10 seconds initially
- Left-align all text; avoid columns for key content

## 2. ATS Optimization

### Why It Matters

Over 98% of Fortune 500 companies use an ATS to filter resumes before a human ever sees them. Modern ATS systems use NLP (natural language processing) and semantic matching — not just exact-keyword lookup — but formatting errors still cause parsing failures regardless of content quality.

### Formatting Rules

- **Layout:** Single-column. No tables, text boxes, sidebars, or columns.
- **Font:** Arial, Calibri, Helvetica, or Times New Roman. 10–12 pt body, 14–16 pt headers.
- **Colors:** Black text on white background. Accent colors may be stripped.
- **File format:** `.docx` is the safest. Text-based PDFs are acceptable; scanned-image PDFs are not.
- **Headers/footers:** Do NOT put contact info in headers/footers — ATS parsers often skip them.
- **Graphics/icons:** Avoid. Images, logos, and decorative elements are invisible to parsers.
- **Acronyms:** Always spell out at least once (e.g., "Search Engine Optimization (SEO)").

### Standard Section Headers

Use these exact labels — creative alternatives may be unrecognized:

```
Professional Summary / Summary
Professional Experience / Work Experience
Education
Skills / Technical Skills
Certifications
Projects
```

### Keyword Strategy

**Keyword density sweet spot:** 2–3% of total word count (roughly 10–15 instances per 500 words).

**Where keywords carry the most weight (ranked):**
1. Professional Summary
2. Skills section
3. Job title/role lines in Work Experience
4. Bullet point context in Work Experience

**How to source keywords:**
1. Analyze 10–20 target job descriptions — note repeated terms, required tools, and skill phrases
2. Mirror exact language from the posting (if they say "data visualization," use that phrase, not "visual analytics")
3. Include both the acronym and full form: `AWS (Amazon Web Services)`
4. Build semantic clusters: mix core skill terms + supporting context + action verbs + quantified outcomes
5. Layer in 2025 trend terms where genuine: AI/ML literacy, remote collaboration, privacy/security, ESG

**Avoid:** Keyword stuffing (white text, hidden sections, repetition without context). NLP-powered ATS now penalizes this.

## 3. Section-by-Section Guidance

### Header

```
Full Name (largest text on page)
Job Title (target role or current title)
Email | Phone | City, State | LinkedIn URL | Portfolio/GitHub
```

- No full address — city and state is sufficient
- LinkedIn URL: customize it (linkedin.com/in/firstname-lastname)
- No photo (US standard; some EU markets differ)

### Professional Summary (2–4 lines)

Opens with title + years + specialization. Includes 1–2 metrics. Closes with value proposition. Fully customized per application. See [Section 6](#6-professional-summary).

### Work Experience (Reverse Chronological)

```
Job Title | Company Name | Month YYYY – Month YYYY (or Present)
City, State (or "Remote")
  • Bullet 1
  • Bullet 2
  • Bullet 3 (most recent role: up to 5; older roles: 2–3)
```

- Lead every bullet with a strong action verb
- Never start with "Responsible for"
- Quantify outcomes wherever possible

### Education

```
Degree, Major | University Name | Graduation Year
GPA (only if ≥ 3.5 and graduated within last 3 years)
Relevant coursework, honors, or thesis (optional)
```

### Skills

Categorized list. See [Section 5](#5-skills-section-strategy).

### Certifications / Licenses

```
Certification Name | Issuing Body | Year (or Expiry)
```

### Optional Sections

- Projects (especially for engineers and recent grads)
- Publications / Speaking
- Volunteer Work (if directly relevant)
- Languages (with proficiency level)

## 4. Bullet Point Craft (STAR-K Method)

STAR-K = **Situation → Task → Action → Result → Keywords**. Condense into a single high-impact bullet.

### Formula

```
[Strong Action Verb] + [What You Did] + [How/Method] + [Quantified Result]
```

### Action Verb Tiers

**Leadership/Strategy:** Led, Spearheaded, Directed, Architected, Championed  
**Building/Creating:** Developed, Built, Designed, Launched, Engineered  
**Improving:** Optimized, Streamlined, Overhauled, Transformed, Modernized  
**Growing:** Grew, Scaled, Expanded, Increased, Generated  
**Reducing:** Cut, Reduced, Eliminated, Decreased, Consolidated  
**Analyzing:** Analyzed, Evaluated, Assessed, Investigated, Diagnosed  
**Managing:** Managed, Coordinated, Oversaw, Administered, Prioritized  
**Delivering:** Delivered, Executed, Shipped, Deployed, Implemented  

**Never use:** "Responsible for," "Worked on," "Helped with," "Assisted in"

### Quantification Categories

When you cannot use a hard number, use a proxy:

| What to Measure | Example Metrics |
|---|---|
| Revenue/savings | "$2.4M ARR," "cut costs by 18%" |
| Scale | "50M+ rows," "200-seat team," "12 countries" |
| Speed/efficiency | "reduced cycle time by 40%," "2× faster" |
| Volume | "processed 1,000 requests/day," "100+ clients" |
| Quality | "99.9% uptime," "NPS +22 pts" |
| Time | "delivered 2 weeks early," "within 6-month runway" |

### Examples

**Weak:**
> Responsible for improving customer satisfaction in the support team.

**Strong (STAR-K):**
> Redesigned customer feedback intake workflow (Zendesk + Salesforce), cutting average resolution time from 4.2 to 1.8 days and lifting CSAT from 71% to 94% within 12 months.

---

**Weak:**
> Worked on backend infrastructure.

**Strong:**
> Architected event-driven microservices migration (AWS SQS + Lambda), reducing P99 latency by 62% and eliminating $180K/yr in over-provisioned EC2 spend.

---

**Consolidated STAR-K and Standford GSB rule for AI-native SWE bullets**
### STAR-K raw form

---
[Action] + [What] + [How] + [Result]
---

 **Stanford GSB rule:** *"Make the most interesting fact at the beginning of the bullet; it will entice the reader to read the rest."*

### Stanford-optimized STAR-K:
---
[Result / strongest signal] + by [Action] + through [How]

## 5. Skills Section Strategy

### Structure

```
Technical Skills: Python, SQL, AWS, Docker, Kubernetes, React, TypeScript
Data & Analytics: Tableau, Power BI, Snowflake, dbt, Apache Spark
AI/ML Tools: OpenAI API, LangChain, Hugging Face, prompt engineering
Project Management: Agile/Scrum, JIRA, Confluence, OKR frameworks
Soft Skills: Cross-functional leadership, executive communication, mentorship
```

**Placement by seniority:**
- Entry-level / career changer: place Skills *above* Work Experience
- Mid-level professional: place Skills *after* Work Experience
- Senior/executive: split into "Technical Skills" and "Leadership & Strategy" sections

### What to Include

- Tools named explicitly in job descriptions
- Industry-standard certifications (AWS SAA, PMP, CPA, CFA, CISSP)
- Programming languages with implied proficiency context (not self-rated scales)
- AI literacy (universally valuable in 2025): "Generative AI tools," "prompt engineering," "AI-assisted workflows"

### What to Exclude

- Outdated software (Flash, FrontPage, pre-2010 tools)
- Generic buzzwords without evidence ("team player," "hard worker," "detail-oriented")
- Skills you cannot speak to for 5 minutes in an interview
- Self-rated proficiency bars (ATS-invisible and recruiter-skeptical)

## 6. Professional Summary

### Template

```
[Job Title] with [X]+ years [domain/specialization]. 
[Signature achievement with metric]. 
[Core methodology or differentiator]. 
[Value proposition for target role].
```

### Examples by Role

**Software Engineer:**
> Full-Stack Engineer with 6+ years building scalable SaaS products in TypeScript and Python. Reduced API p95 latency by 55% at Acme Corp by migrating to event-driven architecture. Passionate about developer experience and testable system design; seeking a senior IC role on a product-led growth team.

**Product Manager:**
> Product Manager with 8 years driving 0→1 and growth-stage B2B SaaS products. Led a cross-functional team of 14 to ship a $3.2M ARR billing overhaul in Q3 2024, 6 weeks ahead of schedule. Expert in discovery frameworks, OKR alignment, and data-informed prioritization.

**Data Analyst:**
> Data Analyst with 4 years translating raw PostgreSQL and Snowflake data into executive dashboards that drove $1.1M in cost savings at XYZ Inc. Proficient in Python, dbt, and Tableau; currently pursuing Google Cloud Professional Data Engineer certification.

### Rules

- Customize for every application — use the exact job title from the posting
- Include 1–2 hard metrics in every summary
- Do not use first-person pronouns ("I," "my")
- Keep to 3–4 lines maximum; recruiters spend < 10 seconds here
- Avoid clichés: "results-driven," "passionate about," "dynamic," "synergy"

## 7. AI-Assisted Resume Generation

### Workflow (3-Pass Method)

**Pass 1 — Intelligence extraction**

Feed the job description to the LLM and extract:
```
Prompt: "Here is the job description: [paste JD]. 
Extract: (1) required hard skills and tools, (2) preferred qualifications, 
(3) action verbs used, (4) implied soft skills, (5) company-specific terminology. 
Return a structured list."
```

**Pass 2 — Tailored resume generation**

```
Prompt: "Here is my master resume: [paste resume]. 
Using the extracted criteria from this job description [paste list from Pass 1], 
rewrite my resume to integrate these keywords and skills where they align 
with my ACTUAL experience. Do NOT fabricate skills or experiences. 
Preserve all factual accuracy. Prioritize bullet points that best match 
the role's core requirements."
```

**Pass 3 — Gap analysis and skills plug**

```
Prompt: "Compare my rewritten resume against the job description. 
List any keywords or required skills that are still missing. 
Suggest additions ONLY if they are genuinely relevant to my background. 
Flag anything that would be an overreach to include."
```

### Key Prompts Library

| Goal | Prompt |
|---|---|
| Summary | "Write a 3-line ATS-optimized professional summary for [Job Title] role at [Company] using my resume. Include 1–2 metrics. Avoid first-person pronouns." |
| Bullet upgrade | "Enhance this bullet point to better showcase impact and include keywords from [JD]: [paste bullet]" |
| Keyword gap | "Identify keywords in this JD that are absent from my resume. List only those I could honestly claim." |
| Skills section | "Generate a categorized skills section using my experience and the top hard skills in this JD." |
| Cover letter | "Write a focused, 3-paragraph cover letter for [role] at [company] using my resume and this JD. Tone: confident but not boastful." |

### AI Safety Rules

- **Never accept AI output verbatim** — always review, edit, and verify factual accuracy
- **AI is a people-pleaser** — it will add skills from the JD even if you have none; use the constraint "only if genuinely relevant to my background"
- **Hallucination risk is real** — if the AI adds "3 years of Oracle Database experience" and you have none, that is a liability in interviews
- **Privacy:** Avoid pasting full resume with PII into public LLM endpoints (use a sanitized version or self-hosted model)

### ATS Limitation of AI Tools

LLMs excel at text generation but do not natively understand ATS parsing rules (column layouts, header/footer exclusion zones, font requirements). Always apply ATS formatting rules manually after AI-generated content. AI writes the content; you control the structure.

## 8. Testing and Iteration

### ATS Simulation Tools

| Tool | Focus |
|---|---|
| Jobscan | Keyword match score vs. JD |
| Resume Worded | ATS and readability scoring |
| Skillsyncer | Gap analysis |
| RezScore | Overall strength rating |

### A/B Testing Protocol

Run 20–30 applications per variant over 3–4 weeks:
- Variant A: keyword density ~1.5%; Variant B: ~2.2%
- Variant A: traditional verbs; Variant B: emerging tech terminology
- Track: ATS pass-through rate, view rate, recruiter outreach, interview rate

### Application Tracking Metrics

- < 10% response rate → keyword or format problem
- High views, no outreach → summary or relevance problem
- Interviews but no offers → interview prep problem (not resume)

## 9. Common Mistakes

### Formatting Mistakes

- Using tables, text boxes, or multi-column layouts
- Putting contact info in the header/footer of the Word document
- Using a scanned PDF or image-based resume
- Inconsistent date formatting (mix of "Jan 2023" and "01/2023")
- Employment dates in wrong section (must be clearly visible per role)

### Content Mistakes

- Starting bullets with "Responsible for" or "Helped"
- Generic objective statements ("Looking for a challenging role...")
- Listing skills you cannot defend in an interview
- Omitting measurable outcomes from every bullet
- Using creative section headers ATS won't recognize ("My Journey," "What I Bring")
- Missing exact-match keywords from the JD
- Listing job duties instead of achievements
- Using first-person pronouns anywhere

### Strategy Mistakes

- Sending the same generic resume to every job
- Not customizing the professional summary per application
- Applying to roles before tailoring — the first impression is the ATS score
- Relying solely on AI output without human review
- Claiming credentials or experience you cannot evidence

## 10. Quick Reference Checklists

### ATS Compliance Checklist

- [ ] Single-column layout, no tables or text boxes
- [ ] Standard fonts (Arial, Calibri, Helvetica, Times New Roman)
- [ ] Saved as .docx or text-based PDF
- [ ] Contact info in body, not header/footer
- [ ] Standard section headers used
- [ ] No images, icons, or graphics
- [ ] Acronyms spelled out at least once
- [ ] Keywords from JD present in Summary, Skills, and Experience

### Bullet Quality Checklist

- [ ] Starts with a strong action verb (not "Responsible for")
- [ ] Contains at least one quantified outcome
- [ ] Includes a keyword from the JD
- [ ] No first-person pronouns
- [ ] Under 2 lines when rendered at 11pt font
- [ ] Describes achievement, not just duty

### Per-Application Tailoring Checklist

- [ ] Professional summary updated with exact target job title
- [ ] 1–2 metrics from most relevant role included in summary
- [ ] Top 5 JD keywords present in Skills section
- [ ] Most relevant bullet points promoted to top of each role
- [ ] ATS scan run (Jobscan or equivalent) before submission
- [ ] File named: `Firstname_Lastname_JobTitle_Company.docx`

### AI Prompting Checklist

- [ ] JD fed to LLM for keyword extraction (Pass 1)
- [ ] Master resume + keyword list submitted for tailoring (Pass 2)
- [ ] Gap analysis run to find missing terms (Pass 3)
- [ ] All AI output reviewed for factual accuracy
- [ ] No fabricated skills, titles, or dates in final document
- [ ] ATS formatting applied manually after AI generation

## 11. Stanford Career Center Standard

> Précis extracted from live audit of `A_terminal_precision_v2.docx` (2026-04-10).
> Sources: Stanford Career Education main guide, Stanford GSB Alumni Career Resources, Stanford CAR Method guide, Stanford PhD/Postdoc guide.

Stanford Career Education (`beam.stanford.edu`) publishes one of the most widely cited institutional resume standards. The rules below are drawn directly from their official PDFs and the GSB alumni resource. Where Stanford is stricter than general market guidance, **Stanford wins**.

---

### 11.1 Formatting Minimums (Stanford-Specific)

| Element | Stanford Rule | Notes vs. General Market |
|---|---|---|
| Margins | **No smaller than 1 inch on all sides** | Stricter — market allows 0.5 in minimum |
| Font size | **No smaller than 10 pt** | Same as market floor |
| Length | **1 page; 2 pages if experience is extensive** | Commit fully — not 1¼ pages |
| Paper / file | White or conservative tone; `.docx` or text-based PDF | Same as market |
| Photo | Prohibited (US/Canada) | Same as market |

---

### 11.2 Required Header Fields

Stanford lists these as explicitly required in the header block. All must be present:

```
Full Name              (largest text element on page)
Email address          (personal, not institutional work email)
Phone number           ← REQUIRED; frequently omitted — automatic audit fail
City, State            (full street address optional / discouraged for privacy)
LinkedIn URL           (Stanford GSB explicitly requires this)
GitHub / portfolio     (good-to-have for tech roles)
```

**Stanford GSB addition:** If the resume runs to 2 pages, repeat the name in the header or footer of page 2 so pages cannot be separated.

---

### 11.3 Date Format

**Stanford rule:** every experience entry must include **month and year** — not year alone.

| Correct | Incorrect |
|---|---|
| `June 2025–Present` | `2025–Present` |
| `9/2023–5/2024` | `2023–2024` |
| `Jan 2022–Mar 2023` | `2022–2023` |

This applies to all experience, education, and project entries. Year-only dates are a compliance fail regardless of aesthetic intent.

---

### 11.4 Verb Tense Rules

**Stanford PhD guide:** *"Use either past or present tense as applicable and keep your format consistent."*

| Role Status | Required Tense | Common Mistake |
|---|---|---|
| Current role (`–Present`) | **Present tense** | Using past tense on all bullets regardless |
| Past role (closed date range) | **Past tense** | Mixing tenses within same role |
| Consistent throughout | Same tense across all bullets in a role | Switching mid-list |

**Practical rule:** If the date range ends in "Present," every bullet in that role must use present tense. `Architected` → `Architect`. `Built` → `Build` / `Maintain`.

---

### 11.5 Experience Entry Structure

Each experience entry must contain all four of:

```
1. Job title (exact, accurate)
2. Organization name + city, state
3. Date range — month AND year
4. Bullet descriptions starting with action verbs
```

**Stanford GSB addition for unknown companies:** If the organization is not widely recognized, add a one-sentence description immediately after the company name:

```
Software Engineer — Acme Corp
Early-stage fintech startup building B2B payments infrastructure for SMBs.
San Francisco, CA  |  Jan 2023–Present
```

---

### 11.6 Bullet Point Rules (CAR Method)

Stanford's native framework is **CAR: Context → Action → Result**, equivalent to the STAR-K method in Section 4. Both arrive at the same structure; Stanford's label is simpler.

**Stanford main guide:** *"Do try quantifying results in your descriptions, such as 'Created marketing campaign that increased club membership by 25%.'"*

**Stanford GSB (stricter):** *"Be as quantitative as possible: revenue growth, money saved, market share growth. Make the most interesting fact at the beginning of the bullet."*

| Rule | Standard | Stanford GSB |
|---|---|---|
| Quantified outcome | Recommended | **Required where possible** |
| Result position | End of bullet | **Beginning of bullet** (GSB) |
| Bullet length | ≤ 2 rendered lines | Same |
| Verb repetition | Avoid | Avoid — use varied action verbs |
| Passive language | Banned | Banned ("duties included," "responsible for") |
| Personal pronouns | Banned | Banned (I, me, we, my) |

**Impact-first reordering (Stanford GSB pattern):**

```
❌ Standard order (action first):
Built Memori, a cross-session memory engine with tiered retention and semantic
recall, reducing context loss across long-running AI-assisted delivery.

✅ GSB order (impact first):
Eliminated context loss in long-running agent sessions by engineering Memori —
a cross-session memory engine with tiered retention, semantic recall, and
checkpoint restore across multi-day delivery workflows.
```

**Banned opening phrases:**
- "Responsible for"
- "Duties included"
- "Helped with"
- "Assisted in"
- "Worked on"

---

### 11.7 Professional Summary Rules

**Stanford GSB:** *"Communicates your core brand and competencies. Limit the summary to 4 lines plus bullet points."*

| Rule | Limit |
|---|---|
| Maximum prose length | **4 lines** |
| Optional bullet points | Allowed after the 4 prose lines |
| No first-person pronouns | Required |
| Tailored per application | Required |

Summaries exceeding 4 lines are non-compliant per Stanford GSB guidance regardless of quality.

---

### 11.8 Prohibited Elements

| Element | Rule |
|---|---|
| Personal pronouns (I, me, we, my) | ❌ Prohibited in all sections |
| Personal information (age, marital status, physical description) | ❌ Prohibited |
| Photograph | ❌ Prohibited (US) |
| "References available upon request" | ❌ Prohibited — wastes space; references go on a separate sheet |
| Objective statement | Optional (summary preferred) |
| High school experience | For freshmen/sophomores only |

---

### 11.9 Action Verb Standard

Stanford publishes a categorized action verb list. For SWE/AI-native roles, draw from these categories:

| Category | Stanford-Listed Verbs |
|---|---|
| **Technical** | Assembled, Built, Calculated, Computed, Designed, Engineered, Fabricated, Operated, Programmed, Solved |
| **Leadership** | Administered, Chaired, Directed, Executed, Expanded, Facilitated, Improved, Initiated, Managed, Oversaw, Supervised |
| **Research/Investigation** | Analyzed, Collected, Conducted, Diagnosed, Evaluated, Experimented, Identified, Investigated, Monitored, Reviewed, Tested |
| **Organizing** | Coordinated, Delegated, Established, Negotiated, Planned, Prioritized, Produced |
| **Detail-Oriented** | Compiled, Documented, Enforced, Processed, Recorded, Systemized |
| **Creative** | Conceptualized, Created, Developed, Designed, Generated, Innovated, Integrated, Synthesized |

**Diversity rule:** avoid repeating the same verb more than once across bullets in the same role. `Built` × 3 = automatic flag.

---

### 11.10 Stanford Audit Checklist

Use this against any draft before submission:

**Header**
- [ ] Phone number present
- [ ] Email present (personal, not work)
- [ ] LinkedIn URL present
- [ ] City, State present
- [ ] No street address (optional/discouraged)
- [ ] No photo
- [ ] Name on page 2 header (if 2-page resume)

**Dates**
- [ ] Every experience entry has month + year (not year only)
- [ ] Every education entry has graduation month + year (or expected)

**Tense**
- [ ] Current roles (–Present) use present tense throughout
- [ ] Past roles use past tense throughout
- [ ] No mixing of tenses within a single role

**Bullets**
- [ ] Every bullet starts with a strong action verb (from Stanford's list)
- [ ] No bullet starts with "Responsible for," "Duties included," "Helped," "Assisted," "Worked on"
- [ ] No personal pronouns (I, me, we, my, our)
- [ ] Every bullet has a quantified or directional result (the R in CAR)
- [ ] Highest-impact fact appears at the start of the bullet (GSB rule)
- [ ] No bullet exceeds 2 rendered lines
- [ ] No action verb repeated more than once per role

**Experience entries**
- [ ] Each entry has: title + org name + city/state + month-year range + bullets
- [ ] Unknown companies have a 1-sentence description
- [ ] Hiretalk.io / independent projects have location noted

**Summary**
- [ ] ≤ 4 prose lines
- [ ] No first-person pronouns
- [ ] Tailored to target role/company

**Prohibited**
- [ ] No "References available upon request"
- [ ] No personal information beyond name/contact
- [ ] No photo
